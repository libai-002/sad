# 后端账号注册系统设计方案

您提到的“设计不合理违规，给出合理设计”是一个非常好的切入点。在开发账号系统时，安全性和合规性是重中之重。

我将从**反面教材（常见的不合理设计）**和**最佳实践（合理设计方案）**两个方面为您规划。

## 1. 常见的不合理/违规设计（我们将避免这些）

*   ❌ **密码明文存储**：直接把用户密码存进数据库。如果数据库被黑客拖库，所有用户密码直接泄露。（**极度危险**）
*   ❌ **缺乏参数校验**：允许用户注册空用户名、超长用户名，或者格式错误的邮箱。这会导致数据脏乱甚至系统崩溃。
*   ❌ **缺乏唯一性检查**：允许同一个邮箱注册多次，导致账号混乱。
*   ❌ **返回信息过细**：注册失败时提示“该邮箱已注册”，可能会被黑客利用进行“撞库”攻击（虽然提升体验，但需权衡安全）。
*   ❌ **SQL 注入风险**：手动拼接 SQL 字符串来插入数据。

## 2. 合理设计方案（我们将实现的）

我们将采用 **Spring Boot + JPA + MySQL** 的经典架构，并引入 **Spring Security** 进行密码加密。

### 技术栈准备
*   **Spring Data JPA**: 用于操作数据库（比手动写 SQL 更安全、方便）。
*   **MySQL Driver**: 连接 MySQL 数据库。
*   **Spring Validation**: 用于参数校验（拦截不合理输入）。
*   **Spring Security Crypto**: 用于密码加密（BCrypt 算法）。

### 核心实现步骤

#### 第一步：添加依赖 (pom.xml)
引入 MySQL、JPA、Validation 和 Security（加密部分）的依赖。

#### 第二步：配置数据库 (application.properties)
**【重要】** 这一步需要您提供 MySQL 数据库的连接信息（地址、账号、密码）。如果您暂时没有 MySQL，我们可以先用 **H2 内存数据库** 来模拟，不影响开发进度。

#### 第三步：设计数据模型 (User Entity)
设计合理的数据库表结构，包含字段：
*   `id`: 主键（自增）
*   `username`: 用户名（唯一，非空）
*   `password`: **加密后的**密码（非空）
*   `email`: 邮箱（唯一，非空，符合邮箱格式）
*   `role`: 角色（如 USER, ADMIN）
*   `created_at`: 注册时间

#### 第四步：实现注册逻辑 (Service & Controller)
这是“合理设计”的核心：
1.  **DTO 传输对象**：定义一个 `RegisterRequest` 类专门接收前端参数，而不是直接用数据库实体，防止恶意字段注入。
2.  **参数校验**：使用 `@NotBlank`, `@Email`, `@Size` 等注解，在进入业务逻辑前就拦截掉不合理的数据。
3.  **唯一性检查**：先查询数据库，确保用户名和邮箱未被注册。
4.  **密码加密**：使用 `BCryptPasswordEncoder` 对密码进行哈希加密，再存入数据库。
5.  **统一响应**：无论成功失败，返回标准的 JSON 格式结果。
